Retransmission Strategies

When packets get lost in transit, how does a protocol recover? This tutorial explains the two classic answers — Go-Back-N and Selective Repeat — building up from first principles so that every design trade-off makes intuitive sense.

Section 01

Why Retransmission Exists

Networks are inherently unreliable. A router might be overloaded and silently drop your packet. A cable might have a burst of electrical noise that corrupts data mid-flight. A wireless signal might collide with another device's transmission. Whatever the cause, packets disappear — and when they do, higher-level software still expects to receive them.

Reliable transport is the engineering discipline of making an unreliable network look reliable to applications. The core mechanism is simple in concept: the sender keeps a copy of everything it sends, waits for the receiver to confirm delivery, and resends anything that was not confirmed within a reasonable time.

The confirmation message is called an acknowledgment, or ACK for short. Think of it like signing for a registered letter at your door: until the postman receives your signature, the letter is considered undelivered and he'll try again. In networking, the timer that decides "try again" is the retransmission timer, and the value it waits for before giving up is the timeout.

But efficiency matters, too. Stop-and-wait — sending one packet, waiting for its ACK, then sending the next — works but wastes most of the available bandwidth. The solution is to send multiple packets simultaneously, which is the idea behind the sliding window. The sliding window is a moving buffer of packets that the sender is allowed to have "in flight" (sent but not yet acknowledged) at any moment. Its size is called N.

This tutorial focuses on one crucial question: once you have a sliding window in place, which packets do you retransmit when something goes wrong?

Setting the Stage

The system uses cumulative acknowledgments, meaning each ACK does not just confirm one specific packet but rather says "I have received everything up to and including packet number X." Finally, each individual packet has its own timer; if that timer expires before an ACK arrives, the packet is considered lost.

Now suppose packet 2 is lost in transit. The receiver gets packets 1, 3, 4, and 5 — but because its window says "I'm expecting packet 2 next," packets 3–5 create an awkward situation. There are two very different philosophies for how to handle it.

Key Takeaway

Reliable transport is built on three pillars: the sender keeps unacknowledged data buffered, per-packet timers detect loss, and cumulative ACKs confirm delivery. The open question is what to retransmit when a loss occurs.

Section 03

Go-Back-N: The Pessimistic Strategy

Go-Back-N (GBN) takes a conservative, almost paranoid stance. Its philosophy is: "If I lost packet 2, I might as well assume the receiver can't use anything that came after it either — so I'll retransmit everything in my window starting from the lost packet."

The receiver in GBN is deliberately simple: it only accepts packets in strict order. If packet 3 arrives before packet 2, the receiver discards packet 3 entirely and keeps asking for packet 2 again. This makes the receiver's logic trivial — it only needs a single-packet receive buffer — but it creates a potentially huge amount of unnecessary retransmission.

Consider a window of size N = 4. If the very first packet in the window is lost, all four packets — including the three that arrived perfectly fine — must be retransmitted. It's like returning an entire shipment of boxes to a warehouse because one package in the delivery was damaged, even though the other three are fine.

Diagram — Go-Back-N: packet 2 is lost (window N = 4)

The key characteristic of Go-Back-N is that one loss causes the entire window to be retransmitted. This is sometimes called a "pessimistic" or "conservative" approach because it assumes the worst: anything after the lost packet should be re-sent.

The tradeoff is stark. GBN keeps the receiver's logic dead simple — no out-of-order buffering needed — but it can be extraordinarily wasteful when the window is large and the loss rate is anything above negligible.

Key Takeaway

In Go-Back-N, a single packet loss triggers retransmission of the lost packet and all subsequent packets in the window. The receiver has a window of 1 (accepts only the expected next packet) and discards out-of-order arrivals.

Section 04

Selective Repeat: The Optimistic Strategy

Selective Repeat (SR) takes the opposite philosophy: "If packet 2 is lost but packets 3, 4, and 5 arrived fine, then let's keep those and only retransmit packet 2." It's the difference between replacing one broken item in a shipment versus returning and re-ordering the entire box.

For this to work, the receiver must also have a buffer of size N — it can hold out-of-order packets while waiting for the missing ones to be filled in. When a gap is eventually filled by a retransmission, the receiver delivers a consecutive chunk to the application and slides its window forward.

This is more complex on both sides. The sender now needs to track a timer per packet, independently. The receiver needs to buffer and track potentially N out-of-order packets. However, SR sends far fewer bytes over the wire: if losses are rare, it retransmits only the handful of packets that were actually lost, not everything in the window.

Diagram — Selective Repeat: only packet 2 is retransmitted (window N = 4)

Notice what happened in the diagram: packets 3, 4, and 5 were buffered by the receiver while it waited for packet 2 to be filled in. When packet 2 was retransmitted and arrived, the receiver could deliver the entire contiguous run (2, 3, 4, 5) to the application at once and advance its window significantly.

This buffering at the receiver is the cost of Selective Repeat. You need memory proportional to N on both sides. But in modern systems where memory is cheap and bandwidth is precious, this tradeoff usually favors Selective Repeat — especially when the window is large (necessary on high-bandwidth, high-latency paths like satellite links) and losses are occasional rather than frequent.

Key Takeaway

In Selective Repeat, only the lost packet is retransmitted. The receiver has a window of size N and buffers out-of-order arrivals. This is more bandwidth-efficient but requires more memory and bookkeeping on both ends.

Section 05

Worked Examples: Receiver Window Matters

The slides from CS 144 walk through two illuminating scenarios that illustrate how the receiver's window size determines which strategy is being used — even when the sender's window is the same.

Example 1: Sender Window = N, Receiver Window = 1

Here, the sender can have N = 4 packets in flight simultaneously, but the receiver only accepts one packet at a time (window of 1). This is the Go-Back-N configuration. When a packet is lost, the sender doesn't hear an ACK for it and eventually times out. Because the receiver has been silently discarding every out-of-order packet, the sender must go all the way back to the lost packet and retransmit from there.

In the diagram from the lecture (annotated with instructor handwriting), we see packets 1–4 sent in the initial window. When packet 2 fails, ACK 1 is returned but ACKs 2, 3, and 4 never arrive. The sender's timer fires for packet 2, and it must re-send packets 2, 3, and 4 all over again. The question posed to students is: "Is this Go-Back-N or Selective Repeat?" — and the answer is Go-Back-N, because the receiver window of 1 forces it.

Example 2: Sender Window = N, Receiver Window = N

Now the receiver also has a window of N = 4. Packets 3, 4, and 5 arrive out-of-order (because packet 2 was lost), and the receiver can buffer all of them. The sender only needs to retransmit packet 2. When it arrives, the receiver can fill the gap and send back a cumulative ACK (ACK 5 in the diagram) that acknowledges all five packets at once. The question posed is again "Go-Back-N or Selective Repeat?" — and the answer is Selective Repeat, because the receiver window of N enables buffering.

This second example also illustrates why Selective Repeat can sometimes advance the window dramatically: in one retransmission round-trip, the receiver jumps forward by N positions rather than just 1.

Key Takeaway

The decisive factor is the receiver's window size. A receiver window of 1 means Go-Back-N (discard out-of-order). A receiver window of N means Selective Repeat (buffer out-of-order). Both share the same sender window size N.

Section 06

Side-by-Side Comparison

The table below summarises the key differences between the two strategies across the dimensions that matter most for protocol design.

Dimension Go-Back-N Selective Repeat
Sender window N (up to N packets in flight) N (up to N packets in flight)
Receiver window 1 (only next expected packet) N (buffers out-of-order packets)
On packet loss Retransmit lost packet + all subsequent in window Retransmit only the lost packet
Out-of-order packets Discarded at receiver Buffered at receiver
Receiver memory Low (1 packet buffer) High (N packet buffer)
Bandwidth efficiency Poor under high loss or large N Good even under moderate loss
Implementation complexity Simpler receiver logic More complex receiver bookkeeping
Philosophy Pessimistic ("assume the worst") Optimistic ("trust what arrived")

In practice, modern TCP implementations lean heavily toward Selective Repeat semantics. TCP receivers can send Selective ACKs (SACKs) — a TCP option that tells the sender exactly which packets arrived out-of-order — allowing the sender to retransmit only the holes. Without SACK, TCP falls back to a simplified Go-Back-N-like behaviour.

Key Takeaway

Go-Back-N minimises receiver complexity at the cost of bandwidth; Selective Repeat maximises bandwidth efficiency at the cost of receiver memory. Real-world protocols like TCP implement Selective Repeat via the SACK option, combining the reliability of GBN with the efficiency of SR.

Summary

The Five Things to Remember

This tutorial has covered a single, concrete question — "what do you retransmit when packets are lost?" — and found that the answer has two very different forms depending on how much complexity you're willing to put in the receiver. Here are the five most important ideas to carry away.

01

Sliding Window is the Foundation

Both strategies use a sender window of size N to keep packets "in flight" simultaneously, filling the network pipe efficiently rather than waiting for each individual ACK.

02

Go-Back-N Retransmits the Whole Window

One lost packet causes N packets to be re-sent. The receiver has window size 1 and silently discards everything out-of-order. Simple receiver, wasteful bandwidth.

03

Selective Repeat Retransmits Only the Lost Packet

The receiver buffers out-of-order arrivals and fills gaps when retransmissions arrive. Efficient bandwidth usage, but requires memory proportional to N at the receiver.

04

The Receiver Window is the Key Differentiator

Both strategies can share the same sender window size N. What distinguishes them is the receiver's window: 1 for GBN, N for Selective Repeat.

05

Real TCP Uses Selective Repeat via SACK

Modern TCP adds a Selective Acknowledgment (SACK) option that tells the sender exactly which segments arrived, enabling targeted retransmission just like Selective Repeat.
