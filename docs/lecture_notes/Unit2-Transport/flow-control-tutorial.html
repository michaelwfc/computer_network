When two computers talk over a network, one of them might be much faster than the other. Without a way to slow the sender down, packets pile up, buffers overflow, and data gets lost — silently. This tutorial walks you through the elegant mechanisms that solve this problem: flow control. You'll learn how Finite State Machines model protocol behavior, how Stop-and-Wait ensures reliability at the cost of speed, and how the Sliding Window protocol reclaims that speed while preserving correctness. These same ideas power TCP — the protocol that carries almost all web traffic today.

Section 01

Finite State Machines

Before we can understand how network protocols handle errors and flow, we need a common language for describing them precisely. That language is the Finite State Machine, or FSM. This section introduces FSMs from the ground up — by the end, you'll see why they're the standard tool for specifying how protocols behave.

A Finite State Machine is a model of a system that can be in exactly one of a limited number of states at any given time. Think of a traffic light: it can be green, yellow, or red — never two at once. Those are its states. The word "finite" just means the number of states is countable; you're not dealing with infinite possibilities.

Abstract FSM — three states with transitions

States, Events, and Actions

An FSM consists of three core ingredients. States represent the possible conditions a system can be in. Events are the triggers that cause a state change — maybe a packet arrives, a timer fires, or a user calls a function. Actions are the things the system does when it transitions: send a message, start a timer, deliver data to the application.

Edges drawn between states in a diagram represent these transitions. The label on each edge is written in two parts: the event that causes the transition, and (optionally below it) the action the system takes as it moves. If no action is listed, the system simply moves to the new state and does nothing else.

⚠ Important rule
For any given state, the transition for any specific event must be unique. If the same event could lead to two different next states, the FSM is ambiguous — the system can only be in one state at a time, so the outcome must be deterministic.

What Happens with Undefined Events?

If a system is in a state and receives an event for which there is no outgoing transition defined, the FSM's behavior is undefined. This is a practical concern: incomplete specifications lead to bugs. Protocol designers must decide whether to explicitly specify every transition or leave some undefined and document them in accompanying text.

The Internet Engineering Task Force (IETF), for example, sometimes deliberately leaves certain transitions unspecified — not by accident, but to keep the spec flexible and let implementers discover what matters through real-world usage.

🔑 Key Takeaway

A Finite State Machine is a precise, visual way to describe how a system behaves: what states it can be in, what events trigger transitions, and what actions accompany those transitions. They're powerful because they make implicit behavior explicit — leaving no room for ambiguous "it depends" answers. Every network protocol you'll encounter, including TCP, is formally specified using FSMs.

Section 02

The Flow Control Problem

Now that we have the vocabulary of FSMs, we can tackle the central problem this tutorial is about: what happens when a sender is too fast for its receiver? This section explains why the problem exists and sets up the two solutions we'll explore.

Imagine you have two computers, A and B, connected by a network. Computer A can generate and send data at 500,000 packets per second. Computer B can only process 200,000 packets per second. A is blazing fast; B is merely quick. If A sends at full speed without any coordination, B will be overwhelmed — it can't process packets fast enough, its buffer fills up, and incoming packets start getting dropped on the floor. Data is lost.

This is the flow control problem: ensuring the sender does not overwhelm the receiver. The receiver needs a way to give the sender feedback about how fast it can cope, and the sender needs to respect that feedback.

Two Fundamental Approaches

There are two classic approaches to flow control, and both are fundamental to networking:

Approach Core Idea Trade-off
Stop-and-Wait Send one packet, then pause until the receiver acknowledges it Simple & safe, but slow — like having one conversation at a time
Sliding Window Send multiple packets at once, up to a limit the receiver can handle Efficient & fast, but more complex bookkeeping

The key insight behind both approaches is the same: the receiver gives the sender feedback. The feedback mechanism differs, but the cooperation is essential.

🔑 Key Takeaway

Flow control is about protecting the receiver from being overwhelmed by a fast sender. Rather than letting the sender blast data as fast as possible, the receiver communicates its capacity back to the sender, and the sender must respect that limit. Without flow control, fast networks would ironically cause data loss — speed without coordination is chaos.

Section 03

Stop-and-Wait

Stop-and-Wait is the simplest possible flow control protocol, and understanding it deeply — including its failure modes — is essential before moving to more sophisticated designs. Think of it as a conversation where you speak one sentence, then wait for the other person to say "I heard you" before speaking again.

The Rules of Stop-and-Wait

The protocol has a clean, minimal ruleset: at most one packet is in flight at any time. The sender sends a single packet and then stops. The receiver, upon successfully receiving the data, sends back an acknowledgment (or ACK for short — think of it as a receipt). When the sender gets the ACK, it sends the next packet. If a timer expires before an ACK arrives, the sender assumes the packet was lost and retransmits it.

The Duplicate Problem

Look at the "ACK Delay" scenario above. The ACK arrived so late that the sender timed out and retransmitted the same data. Now the receiver gets two copies of the same packet. Without any way to distinguish between them, it might deliver the same data to the application twice — a serious bug.

The fix is elegant: include a 1-bit counter in both data packets and acknowledgments. The sender alternates the counter value between 0 and 1 for each new packet. The receiver uses this bit to determine whether an incoming packet is fresh new data or a retransmission to discard. This tiny addition solves the duplicate problem completely, under two simplifying assumptions: the network never duplicates packets on its own, and packets are never delayed across multiple timeout intervals.

Stop-and-Wait — Sender FSM (pseudocode)
// Sender starts in "Wait for Data" state (software has data to send)
state WaitForData:
  on softwareSend(packet):
    sendPacket(packet, seqNo)      // seqNo is 0 or 1
    startTimer()
    goto WaitForACK

state WaitForACK:
  on receiveACK(ackNo):
    if ackNo == seqNo:              // correct ACK?
      seqNo = 1 - seqNo            // flip bit: 0→1, 1→0
      cancelTimer()
      goto WaitForData
  on timeout:
    sendPacket(packet, seqNo)      // retransmit same packet
    restartTimer()

This FSM captures the entire sender logic in two states. Notice how seqNo = 1 - seqNo flips between 0 and 1 as each new packet is acknowledged — that single bit is the duplicate-detection mechanism.

Stop-and-Wait — Receiver FSM (pseudocode)
// Receiver waits for packets, sends ACK, delivers to application
state WaitForPackets:
  on receiveData(packet, seqNo):
    if seqNo == expected:           // new data, not a duplicate
      deliverToApp(packet)
      expected = 1 - expected       // advance expected sequence
    sendACK(seqNo)                  // always ACK, even duplicates
    goto WaitForPackets             // always return to same state

The receiver only ever has one state. Crucially, it sends an ACK even for duplicates — this "heals" the case where the original ACK was lost, preventing the sender from retransmitting forever.

The Performance Problem

Stop-and-Wait works correctly, but it's slow. Imagine a connection between San Francisco and Boston — a round-trip time (RTT) of roughly 50 milliseconds. The bottleneck link can carry 10 Mbps (megabits per second). With Stop-and-Wait, you send one packet, then wait 50ms for the ACK. During those 50ms, the link sits completely idle. You're using only a tiny fraction of the available bandwidth.

The problem is the protocol never keeps the "pipe" full. Like a highway where only one car is allowed at a time, most of the road sits empty. This motivates the next protocol.

🔑 Key Takeaway

Stop-and-Wait achieves correctness: with a single 1-bit counter, it handles data loss, ACK loss, and delayed duplicates cleanly. Its fatal flaw is efficiency: one packet in flight at a time means the network pipe is almost always empty. On high-latency links, you're wasting the vast majority of available bandwidth. The 1-bit sequence number is an important pattern — minimal overhead for maximum correctness.

Section 04

Sliding Window

The Sliding Window protocol is a direct generalization of Stop-and-Wait. Instead of waiting for one ACK before sending the next packet, the sender is allowed to have several packets "in flight" simultaneously — up to a limit called the window size. This keeps the network pipe full and dramatically increases throughput. This section covers how the sender and receiver each maintain their own "window" state.

Why "Sliding Window"?

Picture a train of numbered packets: 0, 1, 2, 3, 4, 5, 6 … The sender can send any packets within its current "window" — say, packets 0 through 4. As ACKs arrive, the window slides forward: once packet 0 is acknowledged, the window advances to cover packets 1 through 5. It's like a magnifying glass moving along the number line of packets.

Sliding Window — the window advances as ACKs arrive

Sender State Variables

The sender maintains three key variables at all times. The Send Window Size (SWS) is the maximum number of unacknowledged packets the sender is allowed to have in flight simultaneously — this is the cap negotiated with the receiver. The Last Acknowledgment Received (LAR) is the sequence number of the most recent ACK received; it marks the left edge of the window. The Last Segment Sent (LSS) is the sequence number of the most recently sent packet; it marks the right edge.

The sender must uphold one invariant at all times:

(LSS − LAR) ≤ SWS

In plain English: the gap between the last thing you sent and the last thing acknowledged cannot exceed the window size. When you receive a new ACK, you advance LAR forward, which "opens up" space in the window to send more packets. The sender must also buffer up to SWS packets in memory, because any of them might need to be retransmitted if their ACK never arrives.

Receiver State Variables

The receiver mirrors this structure. It has a Receive Window Size (RWS) — how many out-of-order packets it's willing to buffer. The Last Acceptable Segment (LAS) is the highest sequence number it will accept right now. The Last Segment Received (LSR) is the most recently received sequence number.

(LAS − LSR) ≤ RWS

If an incoming packet's sequence number is less than LAS, the receiver accepts it and sends back an acknowledgment. Crucially, TCP uses cumulative acknowledgments: if packets 1, 2, 3, and 5 have arrived but not 4, the receiver only acknowledges up to 3. It won't acknowledge 5 until 4 arrives, because the application needs data delivered in order. In TCP's notation, an ACK of "4" means "I have everything up to and including 3 — please send 4 next."

Sender logic — enforcing the window invariant
let SWS = 4        // window size: max 4 unacked packets
let LAR = 0        // last acknowledgment received
let LSS = 0        // last segment sent
let buffer = []   // retransmit buffer

function trySend(packet) {
  if (LSS - LAR < SWS) {       // window has space?
    LSS++
    buffer.push(packet)
    transmit(packet, LSS)
    startTimer(LSS)
  } else {
    waitForAck()              // window full — must wait
  }
}

function onAck(ackNo) {
  LAR = Math.max(LAR, ackNo)  // advance left edge of window
  buffer.removeAcknowledged(ackNo)
  trySend(nextPacket)          // window opened up, try sending
}

Each time an ACK arrives, LAR advances and the window "slides" forward, creating room for new packets. The buffer holds all sent-but-not-yet-acknowledged packets, ready for retransmission if a timer fires.

How Much Sequence Number Space Do We Need?

This is a subtle but important question. Sequence numbers must be large enough to avoid ambiguity — the receiver must always be able to tell whether an arriving packet is a new packet or a retransmission of an old one. The required space depends on the relationship between RWS and SWS:

Scenario Required Sequence Numbers Why
RWS = 1 ("Go-Back-N") SWS + 1 One extra to distinguish new data from retransmissions
RWS = SWS 2 × SWS Both sender and receiver buffers can be full simultaneously
General case RWS + SWS RWS packets in ambiguous ACK state + SWS in flight

The "Go-Back-N" protocol, where RWS = 1, means the receiver discards any out-of-order packet and forces the sender to retransmit from the point of loss. This is simple but wasteful. When RWS = SWS, the receiver can buffer all out-of-order packets, and the sender only retransmits truly lost packets — much more efficient.

🔑 Key Takeaway

The Sliding Window protocol solves Stop-and-Wait's efficiency problem by allowing a whole "window" of packets to be in flight at once. Both sender and receiver track three variables (window size, last-acknowledged, last-sent/received) and maintain a simple invariant. The window size determines throughput: larger windows keep high-latency links busy. The sequence number space must be sized carefully — it needs to be at least RWS + SWS to prevent ambiguity between new packets and retransmissions.

Section 05

TCP Flow Control in Practice

Everything you've learned so far comes together in TCP — the Transmission Control Protocol that carries the vast majority of internet traffic. This section shows how TCP implements sliding window flow control using a specific field in its packet header, and briefly introduces how the full TCP connection lifecycle is modeled as an FSM.

The Window Field in TCP Headers

TCP has flow control built directly into its packet format. Every TCP packet carries a window field in its header. The receiver fills this field with its current RWS — how much more data it can accept — and sends it back to the sender with every acknowledgment. This is called receiver window advertisement.

The sender's rule is simple: it can only send data up to LAR + window. If the receiver's window shrinks to zero (its buffer is full), the sender stops transmitting entirely until the window reopens.

Every time the receiver sends an acknowledgment, it also tells the sender: "here's how much more buffer space I have." The sender uses this to dynamically adjust how many bytes it sends. If the receiver's application is consuming data quickly, the window stays large. If the application slows down, the window shrinks, naturally throttling the sender.

The TCP Connection FSM

TCP connection management — how connections are opened and closed — is one of the most famous FSMs in networking. While a complete walkthrough is beyond this tutorial's scope, the key insight is that the three-way handshake (SYN → SYN/ACK → ACK) is modeled as an FSM with states like CLOSED, LISTEN, SYN_SENT, SYN_RECEIVED, and ESTABLISHED.

TCP 3-way handshake — simplified FSM transitions
// Client side (active opener)
CLOSED  → [connect()]  → SYN_SENT    // sends SYN
SYN_SENT → [recv SYN/ACK] → ESTABLISHED // sends ACK

// Server side (passive opener)
CLOSED   → [listen()]    → LISTEN       // no packets sent
LISTEN   → [recv SYN]    → SYN_RECEIVED // sends SYN/ACK
SYN_RECEIVED → [recv ACK] → ESTABLISHED  // connection open!

// Both sides now in ESTABLISHED: data flows freely

The FSM makes it precise: calling connect() sends a SYN and moves the socket to SYN_SENT; receiving a SYN/ACK response completes the client side; receiving the final ACK completes the server side. Before FSMs, these rules lived only in prose — ambiguous and error-prone.

Connection Teardown: The FIN Handshake

Closing a TCP connection is more complex than opening one, because TCP is bidirectional. Either side can "half-close" its direction while still receiving data from the other. When one side calls close(), it sends a FIN packet and enters FIN_WAIT_1. The other side acknowledges and enters CLOSE_WAIT. Both sides eventually exchange FINs and acknowledgments through a sequence of states — FIN_WAIT_2, TIME_WAIT, LAST_ACK — before the connection fully closes.

The TIME_WAIT state deserves a mention: it's a deliberate delay before the socket is fully closed, to ensure any lingering delayed packets from the old connection are flushed from the network before the same port pair could be reused. It typically lasts 2× the maximum segment lifetime (usually around 60 seconds).

🔑 Key Takeaway

TCP implements sliding window flow control through a single window field in every packet header. The receiver dynamically advertises how much buffer space it has; the sender adjusts accordingly in real time. The entire lifecycle of a TCP connection — from CLOSED through ESTABLISHED to TIME_WAIT — is an FSM with 12 states. What once seemed like vague prose ("shake hands, then exchange data, then close") becomes a precise, implementable specification when modeled as an FSM.

Summary

The Five Things That Matter

Flow control protects the receiver. When a sender is faster than a receiver, data gets dropped unless the receiver can signal its capacity back to the sender. The two main strategies — Stop-and-Wait and Sliding Window — both rely on the receiver giving feedback; they differ only in how much data is allowed in flight at once.

FSMs make protocol behavior precise. A Finite State Machine turns fuzzy prose descriptions ("send a packet, wait for acknowledgment") into a deterministic specification: states, events, actions, and transitions. Any ambiguity in an FSM is a bug waiting to happen. TCP's 12-state FSM is the canonical example of this rigor in practice.

Stop-and-Wait trades efficiency for simplicity. Limiting the sender to one packet in flight at a time is easy to implement and reason about. A single 1-bit alternating counter handles duplicate detection. But on any link with non-trivial latency, the network pipe sits mostly idle — which is why this protocol is rarely used alone.

Sliding Window keeps the pipe full. By allowing SWS packets in flight, the sender can continuously fill the link without waiting for individual ACKs. The window "slides" as acknowledgments arrive. Choosing the right window size is a function of bandwidth and round-trip time: window ≥ bandwidth × RTT (the "bandwidth-delay product") to saturate the link.

TCP puts it all together with a window field. The 16-bit window field in every TCP header is the receiver's real-time advertisement of available buffer space. Sender and receiver continuously negotiate the effective flow rate through this field. This is how your browser download adapts to a slow server, and why streaming video buffers when your network is congested — the window is shrinking.
