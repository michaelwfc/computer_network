From first principles to TCP Reno — a complete walkthrough of how the Internet avoids collapse and converges to fair, efficient sharing.

01 What Is Congestion?

Congestion happens when more data arrives at a router than it can forward. The router's queue fills. If the queue overflows, packets are dropped. TCP must detect this and respond.

Intuition Think of a highway on-ramp during rush hour. Cars (packets) pile up because the highway (link) can't absorb them all. The on-ramp queue (router buffer) is the only place to put the overflow.

Time scales of congestion

ScaleExampleMechanism
~100 µsTwo packets arrive at onceMicro-burst collision in a single router queue
~10 msMultiple flows saturate a linkSustained overload, queue grows steadily
~hoursFlash crowd / peak hourToo many users simultaneously
              ┌──────────────┐
In1 (10 Mb/s) ──►│              │
               │  Router      ├──► Out (10 Mb/s, bottleneck)
In2 (10 Mb/s) ──►│  + Queue     │
               └──────────────┘

  In1 + In2 = 20 Mb/s  >  Out = 10 Mb/s
  → queue grows → buffer fills → PACKET DROP
Key insight: congestion is unavoidable — and arguably good Empty queues mean low delay but wasted capacity. Occupied queues mean efficient usage but higher delay. The goal is to keep queues occupied without overflowing them.

02 Why Congestion Happens

The Internet uses packet switching precisely because it makes efficient use of links. Efficiency requires that links are sometimes fully utilized — which means queues fill up.

Additional observations

  • When packets are dropped, retransmissions add more load — making congestion worse.
  • Dropped packets waste all upstream resources consumed before the drop point.
  • We need a definition of fairness to decide how flows should share a bottleneck.
The vicious cycle of pre-Tahoe TCP Congestion → drops → retransmissions → more congestion → more drops → collapse. This actually happened in 1986. The Internet became unusable.

03 Fairness and Max-Min Fairness

Why we need a fairness definition

Consider flows A, B, C sharing a network. Two allocations:

AllocationrArBrCTotalFair?
#10.251.750.752.75❌ B takes almost everything
#20.51.50.52.5✓ Better share

Max-Min Fairness

Definition An allocation is max-min fair if you cannot increase the rate of one flow without decreasing the rate of another flow that already has a lower rate.

On a single link with N flows and capacity C, the max-min fair allocation gives each flow exactly C/N. In a multi-hop network, the calculation is more complex — some flows are constrained by different bottlenecks.

Goals for congestion control

  1. High throughput: keep links busy and flows fast
  2. Max-min fairness: distribute capacity equitably
  3. Fast response: react quickly to changing conditions
  4. Distributed control: endpoints decide, not a central authority

04 Where to Put Congestion Control?

In the Network (Routers)

  • Per-flow queues at each router
  • Fair Queueing (FQ) scheduler
  • Precise, guaranteed fairness
  • Routers track per-flow state
  • Does not scale to the Internet

At the End Host (TCP)

  • Infer congestion from packet loss
  • Adjust sending rate accordingly
  • Routers stay simple (just drop)
  • Scales to billions of flows
  • Relies on correct endpoint behavior
The Internet's choice TCP implements end-host-based congestion control. Intelligence lives at the edges; the network core stays simple. This design scales, but depends on endpoints playing by the rules.

05 AIMD: The Core Algorithm

AIMD = Additive Increase, Multiplicative Decrease. This is TCP's steady-state algorithm for probing available bandwidth.

  If packet received OK:   W ← W + 1/W    (≈ +1 MSS per RTT)
  If packet is dropped:    W ← W / 2      (halve immediately)

The sawtooth is TCP's stable operating point. Senders probe upward with additive increase, then back off sharply on a detected loss. Over many flows, the sawteeth average out to a consistently high link utilization.

06 AIMD with a Single Flow

Sending rate equation

The throughput of a sender with window size W and round-trip time RTT is:

  R = W / RTT

Two operating regimes

ConditionBehaviorProblem
R × RTT > WSender is window-limited; waiting for ACKsACKs arrive in bursts — link may idle
R × RTT = W ✓Pipe exactly full; ACKs evenly spacedNone — this is the optimal state

Buffer sizing

Rule of thumb For a single AIMD flow to keep the link continuously busy, the router buffer must be at least:

Buffer ≥ RTT × C

where C is the link capacity. Too small a buffer → the link goes idle after a drop.

Key observations

  • Window expands/contracts to probe how many bytes the pipe can hold.
  • The sawtooth is the stable, self-regulating operating point.
  • Sending rate is effectively constant if buffers are sized correctly.

07 Multiple Flows and the Throughput Equation

With many flows sharing a bottleneck, the buffer always contains packets from multiple senders. Unlike a single flow, sending rate now varies with window size:

  Single flow:    R = W/RTT(t)  ≈ constant  (RTT also varies with window)
  Multiple flows: R = W(t)/RTT  ∝ W(t)      (RTT stays roughly constant)

TCP Throughput Equation

From geometric analysis of the AIMD sawtooth:
R = √(3/2) / (RTT × √p)
R = per-flow throughput  |  RTT = round-trip time  |  p = packet drop probability

Interpreting the equation

ChangeEffect on throughputImplication
RTT doublesR halvesLong-RTT flows are penalized — a fairness problem
p → 0R → ∞Bounded by flow window and link capacity in practice
p = 1%Very low RAIMD is extremely sensitive to loss rate
RTT unfairness Two flows on the same link — one with RTT = 10 ms, one with RTT = 100 ms — will converge to a 10:1 throughput ratio. AIMD does not achieve max-min fairness when RTTs differ. This is a known limitation.

08 TCP History: The Collapse

YearEvent
19743-way handshake developed by Vint Cerf et al.
1978TCP and IP split into separate layers
1983ARPAnet switches to TCP/IP on January 1
1986Internet suffers congestion collapse — becomes unusable
1987–88Van Jacobson diagnoses and fixes TCP → TCP Tahoe
1990Fast recovery added → TCP Reno

What was congestion collapse?

Pre-Tahoe TCP had no congestion window. On connection establishment, the sender immediately transmitted a full window of segments — potentially overwhelming the bottleneck.

The collapse mechanism 1. Sender blasts full window → most packets dropped at bottleneck
2. TCP retransmits dropped packets → adds more traffic to an already congested network
3. New segments are then also dropped → more retransmissions
4. The network is 100% busy doing useless work. Applications see near-zero throughput.

Van Jacobson's insight was that TCP needed to probe the network slowly, not burst into it blindly. His three fixes — congestion window, better RTT estimation, and self-clocking — are still at the core of every TCP implementation today.

09 Congestion Window and Slow Start

The congestion window (cwnd)

Flow control limits what the receiver can handle. The congestion window limits what the network can handle:

  Sender window = min(flow control window, congestion window)

Two states

StateWhencwnd update per ACKGrowth rate
Slow StartStartup, or after timeoutcwnd += MSSExponential (doubles per RTT)
Congestion Avoidancecwnd ≥ ssthreshcwnd += MSS²/cwndLinear (+1 MSS per RTT)
Slow Start trace:
  RTT 1: cwnd=1, send 1 pkt  → receive 1 ACK  → cwnd=2
  RTT 2: cwnd=2, send 2 pkts → receive 2 ACKs → cwnd=4
  RTT 3: cwnd=4, send 4 pkts → receive 4 ACKs → cwnd=8
  ...  (doubles each RTT until cwnd ≥ ssthresh)

Congestion Avoidance trace (cwnd=8, MSS=1):
  Each of 8 ACKs adds 1/8 → total +1 per RTT (linear)
Why "slow" start? Starting at 1 MSS seems slow — but compared to the old approach of blasting the entire flow window immediately, it's a much gentler probe. The name is relative, not absolute.

The ssthresh variable

ssthresh (slow start threshold) is the boundary. When a loss occurs: ssthresh = cwnd / 2. TCP remembers where trouble started and becomes conservative near that point on subsequent passes.

11 RTT Estimation

Why it matters

RTO too shortRTO too long
False retransmissions → unnecessary slow startsLong idle periods before retransmit
Congestion window shrinks without real congestionPoor utilization on lossy links

Pre-Tahoe: EWMA only

  r   = α × r + (1-α) × m    (EWMA of RTT samples)
  RTO = 2 × r

  Problem: assumes variance is a constant fraction of mean.
  Fails when RTT is high but stable (RTO is wastefully large)
  or low but variable (RTO triggers false timeouts).

TCP Tahoe: EWMA + Variance Tracking

  e   = m - r                  (error between measurement and estimate)
  r   = r + g × e              (update mean, g ≈ 0.125)
  v   = v + g × (|e| - v)      (update variance estimate)
  RTO = r + 4 × v              (mean + 4σ ≈ covers most of the distribution)

  If retransmission itself times out: exponentially back off RTO.

Lesson that generalizes When estimating a retry timer in any networked system, track not just the average delay but also its variance. This principle applies far beyond TCP: HTTP retries, RPC timeouts, distributed system heartbeats.

12 Self-Clocking

Self-clocking is an emergent behavior that arises when TCP sends ACKs as quickly as possible.

Sender              Wide link       Bottleneck (slow)    Wide link         Receiver
  │ ───pkt1─────────────────────►│                                            │
  │ ───pkt2─────────────────────►│ (pkt1 queued)                              │
  │ ───pkt3─────────────────────►│ (pkt2 queued)                              │
  │                               │──►pkt1 (slow)──────────────────────────►│
  │◄── ACK1 ─────────────────────────────────────────────────────────────────│
  │ ─ send pkt4                   │──►pkt2 (slow)──────────────────────────►│
  │◄── ACK2 ─────────────────────────────────────────────────────────────────│
  │ ─ send pkt5                   │──►pkt3 (slow)──────────────────────────►│
  ...

ACKs arrive spaced at the bottleneck rate → sender sends at the bottleneck rate.
Why self-clocking is powerful No explicit rate negotiation is needed. The bottleneck automatically paces ACKs, which pace the sender. TCP only puts a new packet into the network when a packet has left — keeping outstanding data stable and preventing congestion from growing.
Consequence: send ACKs immediately Delayed ACKs (common in some implementations) break self-clocking and degrade throughput. The ACK is a critical timing signal, not just a delivery confirmation.

13 TCP Reno: Fast Retransmit and Fast Recovery

TCP Tahoe's weakness: a triple dup ACK drops cwnd all the way to 1 and enters slow start. This is too aggressive when only one packet was lost and the rest of the network is fine.

Fast Retransmit (in both Tahoe and Reno)

If the sender receives 3 duplicate ACKs, retransmit the missing segment immediately — without waiting for a timeout.

Why 3 dup ACKs signal a loss The receiver is continuing to receive subsequent segments and is acknowledging them. That means the network isn't broken — just one segment is missing. This is very different from a timeout.

Fast Recovery (TCP Reno only)

EventTCP TahoeTCP Reno
Triple dup ACKcwnd = 1, slow startcwnd /= 2, stay in CA
Timeoutcwnd = 1, slow startcwnd = 1, slow start

In steady state with no timeouts, TCP Reno is true AIMD: halve on loss, add on success.

Congestion Window Inflation

While in fast recovery, for each duplicate ACK received:

  cwnd += 1 MSS

Each duplicate ACK proves another segment left the network. TCP can safely inject a new segment in its place. Net effect after receiving all the duplicate ACKs:

  Start of fast recovery:   cwnd = c/2
  After c duplicate ACKs:   cwnd = c/2 + c = 3c/2
  Sender can transmit:      c/2 new segments during the recovery RTT

  On new ACK (retransmit confirmed): deflate cwnd = c/2 (correct value)

14 Why AIMD Converges to Fairness: The Chiu-Jain Plot

Consider two flows A and B sharing a bottleneck of capacity C. Plot their rates as a point (rA, rB).

  rB ▲
     │                  / ← Fair: rA = rB
     │                 /
     │    overload    /
     │   ◄──────────X──────── Efficient: rA + rB = C
     │              │  underload
     │              │
     └──────────────────────► rA

  Ideal operating point: X (intersection — fair AND efficient)

How AIMD moves the point

  • Additive Increase: both flows increase equally → diagonal move (+1, +1) → moves toward the efficient line
  • Multiplicative Decrease: both halve → move along line from point toward origin → changes the ratio of rA:rB
  rB
  ▲              / Fair
  │      t2     /   t4
  │       ●    /    ●
  │      ↗    /   ↗       (AI = diagonal up-right)
  │  t1 ●    /  t3●
  │    \ │  / ↘ /         (MD = toward origin)
  │     \│ /   X ← optimal
  └──────────────────────► rA
AIMD converges to fairness Regardless of starting point, AIMD oscillates around the fair+efficient point. No central coordinator needed. The geometry of additive increase and multiplicative decrease guarantees convergence.

15 Worked Example: AIMD Sawtooth Calculations

Scenario: Alice streams HD video at 10 Mb/s. All packets are 250 bytes (2,000 bits). Minimum ping = 5 ms. Buffer perfectly sized to never go empty. Click each question to reveal the answer.

Q1 What is the minimum (trough) AIMD window?

The minimum RTT of 5 ms occurs when the buffer is empty but the bottleneck link is full.

Bytes in flight = RTT × rate = 5 ms × 10 Mb/s = 50,000 bits = 6,250 bytes
Answer: 6,250 bytes
Q2 What is the maximum (peak) AIMD window?

When both buffer and link are full, RTT doubles to 10 ms.

Bytes in flight = 10 ms × 10 Mb/s = 100,000 bits = 12,500 bytes
(6,250 bytes on link + 6,250 bytes queued in buffer)
Answer: 12,500 bytes
Q3 How big is the router buffer?

Buffer holds one minimum-RTT worth of data at the link rate:

5 ms × 10 Mb/s = 50,000 bits = 6,250 bytes
Answer: 6,250 bytes
Q4 After a drop, how long to recover to the maximum window?

Each packet is 2,000 bits → window grows by 2,000 bits per RTT.

Distance to climb = 12,500 - 6,250 = 6,250 bytes = 50,000 bits
RTTs needed       = 50,000 / 2,000 = 25 RTTs
Average RTT       = (5 + 10) / 2  = 7.5 ms
Time              = 25 × 7.5 ms   = 187.5 ms
Answer: 187.5 ms
Q5 With Australia server (RTT = 250 ms), how big should the buffer be? 
Buffer = min RTT × rate = 250 ms × 10 Mb/s = 2,500,000 bits ≈ 298 KB

Note: 50× larger RTT requires 50× larger buffer. This illustrates how buffer requirements scale with RTT.

Answer: ~298 KB
Q6 With Australia server, how long to recover after a drop? 
Packets to fill buffer = 2,500,000 / 2,000 = 1,250 RTTs
Average RTT            = (250 + 500) / 2    = 375 ms
Time                   = 1,250 × 375 ms     = 468,750 ms ≈ 8 minutes

This dramatically illustrates how AIMD penalizes high-RTT connections. A transatlantic flow loses nearly 8 minutes of recovery time per drop.

Answer: ~468 seconds (~8 minutes)

16 Summary

Pre-Tahoe (1986)

  • No congestion window
  • Blasts full window on connect
  • Poor RTT estimation
  • No self-clocking
  • Caused congestion collapse

TCP Tahoe (1988)

  • Congestion window (cwnd)
  • Slow start
  • Congestion avoidance
  • ssthresh
  • RTT estimation + variance
  • Self-clocking
  • Fast retransmit
  • Fast recovery
  • Window inflation

TCP Reno (1990)

  • Everything in Tahoe
  • Fast recovery
  • Window inflation
  • True AIMD in steady state
  • Stays in CA on dup ACK

Key equations

QuantityFormulaNotes
ThroughputR = √(3/2) / (RTT × √p)Inversely proportional to RTT and √p
Buffer sizeB = RTT × CNeeded to keep link busy during AIMD
Sender windowW = min(cwnd, rwnd)Congestion and flow control combined
RTO (Tahoe)RTO = r + 4vMean + 4× variance

Key principles to remember

  • Congestion is unavoidable — manage it, don't try to eliminate it.
  • AIMD self-regulates to fairness through geometry, not negotiation.
  • RTT penalization is a fundamental AIMD limitation.
  • ssthresh is TCP's memory of where congestion occurred.
  • Self-clocking paces the sender to match the bottleneck — for free.
  • Fast recovery keeps TCP in congestion avoidance, maintaining true AIMD.
